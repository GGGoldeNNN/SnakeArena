# 游戏设计理论知识库

## 核心使命

这是一个**持续生长**的游戏设计理论知识库技能。它的使命是：

1. **吸收** — 从一本本游戏设计书籍中提取核心理论、原则、框架和实践方法
2. **组织** — 将知识按主题分类，建立交叉引用，形成体系化的知识网络
3. **运用** — 在游戏设计任务中，自动调用所有已吸收的理论知识来指导设计决策

## 知识库结构

```
references/
├── INDEX.md                    # 主索引：当前已吸收的所有书籍和知识主题
├── books/                      # 按书籍组织的原始知识提取
│   ├── 书名1-作者.md
│   └── 书名2-作者.md
├── categories/                 # 按主题分类的综合知识
│   ├── 核心玩法与机制设计.md
│   ├── 关卡与空间设计.md
│   ├── 叙事与世界观.md
│   ├── 玩家体验与心流.md
│   ├── 系统与平衡性.md
│   ├── 美学与音效.md
│   ├── 原型设计与测试.md
│   └── 游戏设计思维.md
└── GLOSSARY.md                 # 术语表（跨书籍统一术语定义）
```

## 工作流程

### 一、吸收新书知识（当用户提供书籍时）

当用户给出一本游戏设计相关的书（PDF 或其他格式），按以下步骤操作：

1. **通读全书**，理解核心论点和整体框架
2. **提取核心知识**，写入 `references/books/书名-作者.md`，结构如下：
   - 书籍元信息（书名、作者、出版年份、核心主题）
   - 核心理论 / 框架（每个理论用独立章节记录，包含定义、原理、应用场景）
   - 关键概念与术语
   - 实践方法与工具
   - 经典案例与示例
   - 金句 / 核心观点摘录
   - 与其他书籍的关联与对比
3. **更新分类知识**，将书中内容归类到 `references/categories/` 下对应的主题文件中
   - 如果该主题文件尚不存在，则创建
   - 如果已存在，则在现有框架下补充新内容，并标注来源（`来源：《书名》`）
4. **更新 INDEX.md**，在索引中添加新书的条目，并在相关主题下标注
5. **更新 GLOSSARY.md**，补充新的术语及其定义

### 二、运用知识设计游戏（当用户要求设计游戏时）

当用户要求设计游戏或解决游戏设计问题时：

1. **先读 INDEX.md**，了解当前知识库覆盖的范围
2. **根据问题类型**，读取对应 `references/categories/` 下的主题知识
3. **综合多本书籍的理论**，给出有理论依据的设计建议
4. **在设计输出中标注引用**，说明某个设计思路出自哪本书（例如：`——出自《游戏设计艺术》`）
5. **鼓励综合创新**，不同理论体系之间的交叉往往能产生最有创意的设计方案

#### 设计输出模板

```markdown
## 设计方案：[方案名称]

### 理论依据
- [理论名称] — 出自《书名》
  > 核心观点简述
- [理论名称] — 出自《书名》
  > 核心观点简述

### 设计方案详情
[详细的设计方案，将理论融入具体设计]

### 理论交叉分析
[不同书籍理论之间如何互相印证或补充]

### 可能的迭代方向
[根据理论知识，指出可以进一步探索的方向]
```

### 三、持续优化

- 定期检查知识库中不同书籍之间是否有**矛盾或冲突**的理论观点，整理成「理论对话」笔记
- 当某个分类积累到足够多的内容时，可以提炼出**综合框架**
- 如果用户提供了实践反馈（试玩、测试结果），将这些反馈与理论知识对照记录

## 核心原则

1. **忠实原著** — 提取理论知识时要忠实于原书作者的表达，不歪曲不曲解
2. **体系化** — 不只是摘抄，要把知识点连接成网络，展示理论之间的关系
3. **可操作** — 理论最终要能指导实际的游戏设计决策，避免纯学术空谈
4. **生长性** — 每本新书都在已有基础上生长，不重复造轮子，而是丰富和深化
5. **引用溯源** — 每个知识点都标注来源，便于追溯和交叉验证

## 设计哲学

游戏设计不是一门精确科学，而是一门 craft（手艺）。不同理论家有不同的视角和方法，没有绝对的对错。本知识库的目标不是给出「标准答案」，而是提供**多维度的理论透镜**，让你在设计时能站在巨人的肩膀上，做出更有依据、更有深度的设计决策。
