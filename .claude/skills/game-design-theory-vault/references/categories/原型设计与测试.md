# 原型设计与测试

> 综合多本书籍关于游戏原型制作、迭代设计、玩家测试的理论知识。

---

## 规划与迭代（来自《体验引擎》）

### 规划过多 vs 规划过少

| 问题 | 表现 | 代价 |
|------|------|------|
| 规划过多 | 在不了解情况时做太多计划 | 浪费时间和资源在错误的方向上 |
| 规划过少 | 完全没有方向和目标 | 团队混乱，进度停滞 |

**关键洞察**：规划太多往往比规划太少更常见、更危险，因为：
- **文化习性**：社会鼓励「有计划」的行为
- **与生俱来的自负**：高估自己预测未来的能力
- **治愈性规划**：做计划本身缓解了焦虑感，让人误以为在推进
- **群体规划偏差**：团队讨论会让计划显得更可行
- **后见偏差**：回顾时总觉得事情「显而易见」

### 迭代（Iteration）
快速试错的核心方法。游戏设计本质上是**探索未知领域**，迭代是唯一可靠的探索方式。

> 「真正的游戏设计师并不知道问题的答案，然而他们知道如何通过测试、计划以及分析来找到答案。」

**规划周期**：根据不确定程度调整规划量
- 高度不确定 → 少规划多迭代
- 相对确定 → 可适当增加规划

## 测试协议（来自《体验引擎》）

### 测试方法
- **自己测试**：发现问题但容易有盲区
- **观察测试**：观察玩家实际行为而非听他们怎么说
- **选择测试对象**：目标玩家群体 vs 随机玩家
- **测试规模**：小型频繁测试优于大型偶尔测试

### 询问的技巧
- ❌ 「这个好玩吗？」（玩家会自动编造理由）
- ✅ 观察玩家的**表情、行为、操作犹豫**
- 玩家总是为情感找理由，但这些理由往往是错的

### 灰盒（Graybox）
用简单几何体代替美术资源进行玩法测试。

- **早产型创作**：在玩法未定型前投入美术资源 → 极大浪费
- **灰盒评估技巧**：在灰盒阶段评估核心玩法的可行性
- **剧本象征**：用象征性占位代替最终内容

## 品质的矛盾性（来自《体验引擎》）

高品质不是一蹴而就的，而是在迭代中逐渐打磨出来的。

- **相信迭代**：每次迭代都让游戏变得更好
- **机缘巧合**：好的设计往往不是「计划」出来的，而是在迭代中「发现」的
- **幻想的误区**：设计师脑中的完美游戏和实际做出来的永远是两回事

## 依赖管理（来自《体验引擎》）

### 依赖堆栈（Dependency Stack）
游戏开发的层级依赖关系：
1. 核心玩法（最底层，必须先确定）
2. 系统层
3. 内容层
4. 美术/音效层（最上层）

- **层叠式不确定性**：底层的不确定性会向上传递放大
- **核心玩法优先**：先确定核心玩法再构建上层内容
- **设计 backlog**：管理待做任务，区分优先级

## 创造知识的方法（来自《体验引擎》）

- **沉思**：静心思考
- **研究**：调研已有知识和竞品
- **美术方法**：通过可视化表达探索想法
- **头脑风暴**：发散式创意生成
- **书面分析**：把想法写下来理清思路
- **辩论**：通过不同观点碰撞得出真相
- **测试**：用实际数据验证假设
- **指标**：用数据衡量游戏表现

## 迭代设计流程（来自《游戏设计艺术》）

### 标准迭代循环

```
理念 → 原型 → 测试 → 评估 → 修改 → 再测试...
```

- **闭环**：每一轮迭代都应该是一个完整的闭环
- **快速**：越快越好，让每次迭代周期尽可能短
- **从粗糙到精细**：先用最粗糙的方式测试核心概念

### 设计文档方法

- 游戏设计文档（GDD）应是**活文档**，持续更新
- 一次只写够用的内容——不要写太多没人读的细节
- 推荐使用 Wikis 或数字文档方便持续更新

### 原型演进路线

1. **情绪板（Mood Board）** — 收集参考图片和感觉
2. **纸上原型** — 最快的方式测试核心玩法
3. **可玩原型** — 数字版快速实现
4. **内部测试** → **外部测试** → **公开测试**

## 游戏测试（来自《游戏设计艺术》）

### 测试五大原则

1. **尽早测试** — 只要有个可玩的东西就测试
2. **频繁测试** — 每次小改动都测试
3. **观察为主** — 看玩家做什么，而不是听他们说什么
4. **寻找问题而非确认** — 测试的目的是发现问题，不是确认自己做得对
5. **沉默观察** — 不要引导玩家，看他们自然行为

### 经典测试透镜

> **透镜 #68：测试** — 玩家为什么会卡住？他们犯了什么错？他们误解了什么？他们注意到了什么？他们对什么感到沮丧？对什么感到兴奋？

### 测试的误区

| 误区 | 问题 |
|------|------|
| 让朋友测试 | 他们不好意思说真话 |
| 自己测试 | 你太熟悉了看不到问题 |
| 只测试一次 | 不同玩家遇到不同问题 |
| 提前告诉玩家怎么玩 | 测试的是教学不是游戏 |
| 问「好玩吗」 | 玩家会编造理由 |

## 设计师的成长（来自《游戏设计艺术》）

1. **多玩游戏** — 分析每个游戏，问自己「为什么这个设计有效/无效？」
2. **多读多学** — 跨学科学习（心理学、人类学、建筑学、音乐…）
3. **多做原型** — 想法不值钱，实现才是关键
4. **接受反馈** — 学会不防御地听取批评
5. **迭代** — 永不满足，永不停歇

## 以玩法为核心的设计流程（来自《游戏设计梦工厂》）

Tracy Fullerton 提出的 **Playcentric Design** 是本书的灵魂方法论。

### 核心原则
- 从第一天起就把玩家体验放在核心位置
- 通过原型快速测试设计假设
- 让真实玩家尽早参与测试
- 每次迭代都基于玩家反馈进行修正

### 原型五大原则（来自《游戏设计梦工厂》）
1. **每个原型回答一个问题** — 明确测试假设
2. **快速而粗糙** — 越快越好，越丑越好
3. **忘记质量** — 目标是获取信息，不是漂亮
4. **不要太留恋** — 做好丢弃的准备
5. **无需交互** — 静态草图也是好原型

### 纸上原型技术（来自《游戏设计梦工厂》）
- **卡牌原型**：用纸片模拟游戏元素
- **棋盘原型**：在地图上移动标记
- **角色扮演原型**：真人扮演游戏系统
- **节拍器方法**：用节拍器控制回合节奏
- **案例**：《光环》纸上原型——坐标纸地图+纸片角色+节拍器

### 软件原型类型

| 类型 | 测试目标 |
|------|----------|
| 技术原型 | 引擎/性能限制 |
| 玩法原型 | 核心机制趣味性 |
| 美学原型 | 视觉风格可行性 |
| 界面原型 | 交互流程清晰度 |

### 游戏测试全流程（来自《游戏设计梦工厂》）
1. **准备** — 确定测试目标、准备环境
2. **招募** — 找到目标受众（不要只找朋友）
3. **观察** — 沉默观察，不引导不提示
4. **记录** — 关键时刻、困惑点、情绪变化
5. **询问** — 测试后问开放性问题
6. **分析** — 找模式，区分普遍问题和个别问题

### 测试类型与阶段

| 类型 | 目的 | 阶段 |
|------|------|------|
| 概念测试 | 验证概念吸引力 | 概念阶段 |
| 可用性测试 | 发现界面/操作问题 | 原型阶段 |
| 玩法测试 | 评估核心玩法趣味性 | 迭代阶段 |
| 平衡性测试 | 检查数值和策略 | 后期阶段 |
| 回归测试 | 确认修改没引入新问题 | 每次修改后 |

## 从新书增加的原型与测试知识

### 设计文档写作（来自《通关！游戏设计之道》）

**单页说明书模板**：
- 篇幅限制：300字以内
- 核心目标：5秒内传达游戏的独特性
- 包含要素：
  - 游戏名称和定位（一句话描述）
  - 目标平台和受众
  - 核心玩法的一句话概括
  - 独特的卖点（USP）
  - 参考游戏（如果像某某游戏+某某游戏的结合）

**GDD（Game Design Document）核心10章节**：
1. **游戏概述** — 游戏名称、定位、目标平台
2. **核心玩法** — 玩家做什么、怎么操作
3. **游戏系统** — 主要规则和机制
4. **角色设计** — 玩家角色和NPC
5. **关卡设计** — 世界结构、关卡流程
6. **叙事** — 故事背景、剧情概要
7. **界面设计** — HUD、菜单、操作映射
8. **美术风格** — 视觉方向、概念参考
9. **音效设计** — 音乐风格、音效需求
10. **技术规格** — 引擎、性能目标、工具链

**写作原则**：
- GDD是**活文档**，随着开发进展持续更新
- 先写核心部分，细节逐步补充
- 每一个章节都要问「这如何服务于核心体验？」

### 数值验证方法（来自《平衡掌控者：游戏数值战斗设计》）

**5种验证手段对比**：

| 方法 | 时机 | 优点 | 缺点 |
|------|------|------|------|
| **静态验算** | 设计阶段 | 低成本、快速验证基础公式 | 无法发现动态问题 |
| **本地测试** | 开发阶段 | 可控环境、便于调试 | 样本量小、场景局限 |
| **批量测试** | 开发中期 | 模拟大量战斗、发现极端情况 | 自动化脚本维护成本 |
| **Beta测试** | 上线前 | 真实玩家行为、大数据量 | 反馈周期长、变量多 |
| **全量数据** | 上线后 | 最真实数据、最大样本 | 问题已影响玩家体验 |

**验证流程建议**：
- 设计阶段：先做静态验算，确保公式正确
- 开发早期：本地模拟测试，发现明显不平衡
- 开发中期：批量自动化测试，验证边缘情况
- 上线前：Beta测试，观察真实玩家行为
- 上线后：持续数据监控，迭代优化
